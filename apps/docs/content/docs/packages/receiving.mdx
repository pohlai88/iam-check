## Package README

Org-scoped goods-receipt lifecycle for Afenda-Lite: draft, line capture, discrepancy record/resolve, post, draft cancel, and posted reverse. The package owns `goods_receipt`, `goods_receipt_line`, and `receiving_discrepancy` mutations; schemas remain in `@afenda/db`.

Every mutation requires `organizationId`, `actorUserId`, `correlationId`, and `idempotencyKey`. V1 supports `purchase_order` receipts only via typed `source: { kind: "purchase_order", purchaseOrderId }`. Warehouse and item references resolve through `MasterLookupPort` and must be active. Posting, cancellation, and reverse require `expectedVersion`.

`purchase_order` create/post require an injected `PurchaseOrderReceivingQueryPort`. Create and post reject non-posted / missing / cross-org POs. Post enforces accepted quantity against Receiving-owned posted accepted totals plus PO over-receipt tolerance (atomic relative to Receiving rows).

`postGoodsReceipt` requires injected `InventoryCommandOptions` and posts stock with `movementType: "receipt"` using **accepted** line quantities. Inventory application status is tracked on the receipt (`pending` / `applied` / `failed`). Receiving does not write `stock_*` tables and does not import `@afenda/purchasing`.

`cancelGoodsReceipt` accepts **draft only**. Posted corrections use `reverseGoodsReceipt` (linked compensating receipt + inventory `createReversalMovement`).

Living consumers are `apps/web` server adapters when wired by the composition root.

## Invariants

- Every Receiving record belongs to exactly one organization.
- Purchase Order, item, UoM and warehouse references must belong to that organization.
- Purchase Order receipts require a posted and currently receivable Purchase Order.
- Source eligibility is revalidated when posting.
- Combined posted accepted quantity cannot exceed remaining quantity plus tolerance.
- Posted receipts are immutable; draft receipts may be cancelled.
- Posted corrections use linked reversal receipts.
- Inventory receives only accepted quantities.
- Discrepancies do not directly mutate Purchasing, Inventory or Payables records.
- Receiving never writes stock tables.
- Every command is authorized, idempotent and concurrency-safe.
- Receipt, audit and outbox effects commit or roll back together.

### Runbook scenarios

| Scenario | Response |
|----------|----------|
| PO closes after receipt draft creation | Post fails with PO closed; cancel draft or wait for reopen |
| Concurrent over-receipt attempt | Second post fails on Receiving-owned accepted sum + tolerance |
| Inventory handler unavailable | Receipt stays posted with `inventory_application_status=failed`; list via `listReceivingInventoryExceptions` |
| Duplicate receipt-posted event | Inventory create uses idempotency keys; post command replay returns original |
| Incorrect posted receipt | `reverseGoodsReceipt` + inventory compensating movement |
| Discrepancy found after posting | Record as open addendum; resolve without rewriting posted quantities |
| Partial Inventory application | Exception list shows pending/failed; re-drive ops after inventory recovery |
| Cross-organization source rejection | Create/post returns NOT_FOUND for foreign PO |
| Failed or missing outbox delivery | Transaction rolls back receipt create/post/cancel/reverse |
